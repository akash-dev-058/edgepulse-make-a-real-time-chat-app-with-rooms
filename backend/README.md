# RealTimeRoomChat Backend

## Overview

This repository contains the **FastAPI** backend for the RealTimeRoomChat application. It provides:
- JWT based authentication (registration & login)
- CRUD operations for chat rooms with membership management
- Persistent message storage in PostgreSQL
- Real‑time message delivery via WebSockets backed by Redis Pub/Sub for horizontal scaling
- Health check endpoint and comprehensive error handling
- Dockerised development environment (PostgreSQL, Redis, FastAPI)
- Automated migrations with Alembic and a seed script for local development
- Unit and integration tests using **pytest** and **httpx**

## Quick Start

bash
# Clone the repo
git clone <repo-url>
cd <repo-dir>

# Copy example env and adjust values if needed
cp .env.example .env

# Build and run containers
make dev


The API will be reachable at `http://localhost:8000`. Swagger UI is available at `http://localhost:8000/docs`.

## Environment Variables

| Variable | Description | Example |
|----------|-------------|---------|
| `DATABASE_URL` | Async PostgreSQL DSN | `postgresql+asyncpg://user:pass@db:5432/chatdb` |
| `REDIS_URL` | Redis connection URL | `redis://redis:6379/0` |
| `JWT_SECRET` | Secret key for signing JWTs (minimum 32 chars) | `super_secure_random_string_min_32_chars` |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Access token lifetime | `30` |
| `CORS_ORIGINS` | Comma‑separated list of allowed origins | `http://localhost:5173,http://localhost:3000` |
| `SENTRY_DSN` | Optional Sentry DSN for error reporting | *(empty for dev)* |

## Production Deployment

The `Dockerfile` builds a minimal image. Use the provided `nginx.conf` as a reverse proxy in front of the FastAPI container, ensuring the `Upgrade` header is passed for WebSocket connections.

## Security Considerations

- All passwords are hashed with **bcrypt** (cost 12).
- JWTs are signed with a strong secret and validated on every request and WebSocket handshake.
- Rate limiting is applied to the login endpoint (5 attempts per minute per IP).
- CORS is locked down to the origins defined in `CORS_ORIGINS`.
- Input validation is enforced both at the Pydantic schema level and server‑side.
- Redis is used only for pub/sub; no sensitive data is stored there.
